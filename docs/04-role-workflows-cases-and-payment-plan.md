# 역할별 접속환경, 사건/파일 구조, 회원가입/결제 UX 계획

## 1. 추가 의견 요약

현재 서비스는 단순히 "음성파일 1개를 올려 녹취록 1개를 받는 구조"로 보면 안 된다.

실제 의뢰 흐름은 더 복잡하다.

```text
의뢰인 1명
-> 여러 사건
-> 사건별 여러 음성파일
-> 파일별 또는 구간별 녹취 의뢰
-> AI 탐색/전체 녹취/지정 구간 녹취/속기사 검수
-> 사건 단위 납품
```

따라서 DB와 화면은 반드시 `회원 -> 사건 -> 파일 -> 작업 -> 납품` 구조로 설계해야 한다.

## 2. 역할별 접속환경 분리

의뢰인, 관리자, 속기사는 업무 목적이 다르므로 같은 화면을 공유하면 안 된다.

### 의뢰인 접속환경

목표는 쉽고 빠른 의뢰다.

주요 화면:

1. 로그인/간편 의뢰
2. 내 사건 목록
3. 사건 생성
4. 음성파일 업로드
5. 의뢰 방식 선택
6. AI 탐색 결과 확인
7. 구간 선택 또는 AI 추천 의뢰
8. 견적 확인
9. 결제
10. 진행 상태 확인
11. 최종 파일 다운로드

의뢰인 화면은 고령층도 쓸 수 있어야 하므로 복잡한 관리 메뉴를 숨긴다.

### 관리자 접속환경

목표는 전체 운영 통제다.

주요 화면:

1. 전체 사건 목록
2. 신규 접수 목록
3. 결제 대기/결제 완료 목록
4. AI 탐색 결과 확인
5. 견적 조정
6. 속기사 배정
7. 작업 상태 관리
8. 최종본 검수
9. 납품 처리
10. 환불/취소/고객 문의 관리
11. 매출/정산 관리

관리자는 사건, 파일, 작업, 결제, 납품을 모두 볼 수 있어야 한다.

### 속기사 접속환경

목표는 빠른 검수와 최종 작성이다.

주요 화면:

1. 내 배정 작업
2. 작업 상세
3. 사건 설명/의뢰 요청사항
4. 선택된 음성파일과 구간
5. 오디오 플레이어
6. AI 초안 편집기
7. 검토 필요 구간 목록
8. 화자/신뢰도/청취불능 플래그
9. 최종 녹취록 제출

속기사는 결제/고객 개인정보 전체/매출 정보를 보지 않도록 제한한다.

## 3. 접속 URL 구조

초기에는 하나의 PWA 안에서 역할별 라우팅을 분리한다.

```text
/                  고객용 메인
/login             로그인
/orders/new        새 의뢰
/cases             내 사건 목록
/cases/{caseId}    사건 상세
/status/{orderId}  진행 상태

/admin             관리자 대시보드
/admin/cases       관리자 사건 목록
/admin/orders      관리자 작업 목록
/admin/payments    결제 관리

/work              속기사 작업 목록
/work/{jobId}      속기사 작업 화면
```

운영이 커지면 관리자와 속기사 앱을 별도 서브도메인으로 분리할 수 있다.

```text
app.example.com      의뢰인
admin.example.com    관리자
work.example.com     속기사
```

MVP에서는 하나의 앱에서 role-based access control로 시작한다.

## 4. 사건 중심 구조

의뢰인은 하나의 녹음파일만 올리지 않을 수 있다.

예시:

```text
사건 A: 임금 미지급
  - 통화녹음 1.m4a
  - 통화녹음 2.m4a
  - 현장녹음.mp3

사건 B: 환불 분쟁
  - 상담녹음.mp3
```

따라서 `case` 개념이 반드시 필요하다.

### 사건에 저장할 정보

```text
case_id
customer_id
title
case_type
purpose
description
opposing_party_name
status
created_at
updated_at
```

`case_type` 예시:

```text
계약
입금
환불
임금
폭언/협박
가족/개인분쟁
회사/노무
법원/기관 제출
기타
```

## 5. 파일 구조

파일은 사건에 귀속된다.

```text

```

필드:

```text
file_id

customer_id
original_filename
storage_key
mime_type
size_bytes
duration_seconds
recorded_at
file_label
upload_status
created_at
```

`file_label` 예시:

```text
1차 통화
2차 통화
현장 녹음
상담 녹음
증거 파일 1
```

## 6. 의뢰 방식

의뢰인은 파일마다 다른 방식으로 의뢰할 수 있다.

### 의뢰 방식 A: 전체 파일 녹취

```text
이 파일 전체를 녹취록으로 작성해주세요.
```

사용자가 구간을 모를 필요가 없다.

### 의뢰 방식 B: 알고 있는 구간만 녹취

```text
03:10~07:45 구간만 녹취해주세요.
```

구간 입력 UI가 필요하다.

### 의뢰 방식 C: 필요한 구간을 모르겠음

```text
어느 부분이 중요한지 모르겠어요. AI가 찾아주세요.
```

무료 미리보기와 4,900원 AI 탐색 흐름으로 이동한다.

### 의뢰 방식 D: 여러 파일에서 관련 구간 찾기

```text
여러 파일 중 환불 관련 대화만 찾아주세요.
```

이 기능은 매우 가치가 높지만 난이도가 있으므로 2차 MVP로 둔다. 다만 DB는 처음부터 지원 가능하게 설계한다.

## 7. 작업 단위

`case`와 `file`만으로는 부족하다. 실제 속기사에게 배정되는 단위는 `work_order` 또는 `transcription_request`여야 한다.

예시:

```text
사건: 임금 미지급
파일: 통화녹음 1
의뢰: 04:20~07:10 구간 녹취
작업: 속기사 A에게 배정
납품: DOCX/PDF
```

### transcription_requests

```text
request_id
case_id
customer_id
request_type
status
pricing_status
estimated_price
final_price
payment_status
created_at
```

`request_type`:

```text
full_file
known_range
ai_find_range
multi_file_ai_find
summary_only
legal_submission
```

### transcription_request_files

하나의 의뢰가 여러 파일을 포함할 수 있으므로 중간 테이블이 필요하다.

```text
id
request_id
file_id
selected_start_ms
selected_end_ms
selection_reason
```

## 9. 회원가입 UX

의뢰인은 회원가입에서 막히면 이탈할 수 있다. 따라서 가입을 처음부터 강제하지 않는 것이 좋다.

### 추천 방식

```text
1. 파일 업로드 먼저
2. 무료 미리보기
3. 결과를 저장하려면 휴대폰/이메일 인증
4. AI 탐색 결제 또는 최종 의뢰 전 계정 생성
```

즉, 가입보다 의뢰 시작을 먼저 하게 한다.

### 로그인 방식

MVP 추천:

```text
휴대폰 번호 + 인증번호
이메일 + 인증 링크
```

비밀번호 방식은 고령층에게 부담이 될 수 있다. 가능하면 비밀번호 없는 로그인을 고려한다.

### 회원/비회원 구분

초기에는 아래처럼 간다.

```text
비회원: 업로드, 무료 미리보기 가능
간편회원: AI 탐색 결제, 사건 저장, 진행상태 확인 가능
정회원: 반복 의뢰, 과거 사건 조회, 파일 보관함 가능
```

## 10. 결제창 안내 컨셉

결제는 두 번 발생할 수 있다.

```text
1차 결제: 4,900원 AI 탐색비
2차 결제: 최종 녹취록 작성비
```

이때 1차 결제를 "AI 사용료"라고만 표현하면 약하다.

무료 미리보기는 비용 누수를 막기 위해 AI 요약을 제공하지 않는다. 무료 단계에서는 파일명, 파일 길이, 업로드 성공 여부, 지원 가능한 의뢰 방식 정도만 보여준다. STT, GPT 요약, 시간대별 주제 분석은 4,900원 AI 탐색 결제 이후에만 실행한다.

즉, 4,900원은 모든 의뢰인에게 강제하는 비용이 아니라 `필요한 구간을 모르는 의뢰인`을 위한 선택 옵션이다.

### 의뢰 방식별 결제 분기

```text
전체 파일 녹취를 원하는 의뢰인
-> AI 탐색 없이 전체 길이 기준 견적

이미 구간을 아는 의뢰인
-> AI 탐색 없이 선택 구간 기준 견적

필요한 구간을 모르는 의뢰인
-> 4,900원 AI 탐색 결제
-> AI가 주요 구간을 찾고 더 저렴한 구간 녹취 견적 제안
```

### 4,900원 옵션의 핵심 메시지

```text
전체 파일을 맡기면 비용이 커질 수 있습니다.
AI가 먼저 중요한 대화 구간을 찾아드리면, 필요한 부분만 녹취 의뢰하여 최종 비용을 줄일 수 있습니다.
```

### 권장 표현

```text
AI가 녹음 내용을 먼저 살펴보고, 필요한 대화 구간과 예상 녹취 비용을 찾아드립니다.
```

### 4,900원 결제창 문구

```text
AI 대화 탐색

녹음 전체를 듣지 않아도 주요 대화 내용을 시간대별로 확인할 수 있습니다.
AI가 입금, 환불, 계약, 폭언/협박 등 주요 구간을 찾아드립니다.

포함 내용:
- 시간대별 대화 요약
- 주요 주제 자동 분류
- 추천 녹취 구간
- 예상 녹취 비용

결제금액: 4,900원

[AI 대화 탐색 시작하기]
```

### 최종 녹취 결제창 문구

```text
최종 녹취록 작성 의뢰

선택하신 구간을 속기사가 검수하여 최종 녹취록으로 작성합니다.

선택 파일:
통화녹음 1.m4a

선택 구간:
04:20~07:10

작성 방식:
법원/기관 제출용

예상 납품:
영업일 기준 1~2일

최종 결제금액:
58,000원

[녹취록 작성 의뢰하기]
```

## 11. 결제 흐름 선택지

모든 사용자를 AI 탐색 결제로 보내면 안 된다. 세 가지 진입로를 제공한다.

```text
[전체 녹취 바로 의뢰]
[알고 있는 구간만 의뢰]
[어디가 중요한지 AI가 찾아주세요]
```

이렇게 해야 사용자의 상황별 이탈을 줄일 수 있다.

첨부 요금표 기준으로 결제 단계에서는 파일 업로드 후 재생시간을 자동 분석하고, 전체 작성 기본요금표에 따라 즉시 견적을 보여준다. 기존 고객 200명 전제가 있으므로, 1차 MVP에서는 신규 고객 설득보다 기존 고객이 전화/카톡 대신 링크로 파일을 올리고 바로 결제하게 만드는 흐름을 우선한다.

자세한 가격 계산식과 자동 견적 정책은 `docs/05-pricing-and-payment-automation-plan.md`를 기준으로 한다.

## 12. 여러 파일 의뢰 UX

사건 생성 후 파일을 여러 개 올릴 수 있게 한다.

화면 예시:

```text
사건명: 임금 미지급 관련 통화

올린 파일
1. 2026-06-01 통화.m4a  10분 21초
2. 2026-06-03 통화.m4a   7분 12초
3. 현장녹음.mp3         18분 02초

무엇을 원하시나요?

[전체 파일 녹취]
[파일별 필요한 구간을 직접 선택]
[AI가 중요한 구간을 찾아주기]
```

여러 파일 AI 탐색은 비용을 파일 수/총 길이에 따라 다르게 계산해야 한다.

## 13. 관리자 운영에서 필요한 추가 기능

관리자에게는 사건 단위와 작업 단위를 둘 다 보여줘야 한다.

```text
사건 목록: 고객 분쟁/제출 목적 기준
작업 목록: 속기사에게 배정할 실제 녹취 작업 기준
```

관리자 화면 필수 필터:

```text
신규 사건
AI 탐색 결제 완료
견적 확인 대기
최종 결제 완료
속기사 배정 대기
작업중
최종 검수 대기
납품 완료
취소/환불
```

## 14. 속기사 운영에서 필요한 추가 기능

속기사는 사건 전체를 이해할 정도의 요청사항은 봐야 하지만, 불필요한 개인정보와 결제 정보는 숨긴다.

속기사에게 보여줄 정보:

```text
사건 유형
제출 목적
요청사항
작업 파일
작업 구간
AI 초안
검토 필요 플래그
납품 양식
마감 시간
```

숨길 정보:

```text
고객 연락처 전체
결제 금액 상세
관리자 내부 메모
다른 사건 파일
```

## 15. 놓치면 안 되는 정책

1. 사건별 파일 보관 기간
2. 고객 삭제 요청 처리
3. AI 탐색비 환불 기준
4. 최종 녹취 결제 후 환불 기준
5. 법원 제출용 문서 책임 범위
6. AI 추천 구간은 참고용이라는 고지
7. 여러 파일 업로드 시 총 분석 비용 고지
8. 속기사 접근 권한과 다운로드 제한

## 16. 장기 확장: 사건 자료 종합 관리

이 프로젝트는 초기에는 녹취록 의뢰 서비스로 시작하지만, 장기적으로는 사건 단위 증거자료 관리 서비스로 확장될 수 있다.

사건에는 음성파일뿐 아니라 아래 자료가 함께 들어올 수 있다.

```text
이미지
동영상
카카오톡 대화 캡처
문자 내용
PDF
계약서
입금내역
진단서/확인서
기타 증거자료
```

따라서 DB와 저장소 구조는 처음부터 `case_files`를 음성 전용으로 너무 좁게 만들지 않는다.

권장 구조:

```text
case_assets
```

필드 예시:

```text
asset_id
case_id
customer_id
asset_type
original_filename
storage_key
mime_type
size_bytes
asset_label
description
uploaded_at
```

`asset_type` 예시:

```text
audio
video
image
chat_capture
sms_capture
document
other
```

음성 녹취 작업이 필요한 파일은 `case_assets` 중 `asset_type=audio` 또는 `video`인 자료를 `transcription_request_files`에 연결한다.

장기적으로는 사건 상세 화면을 다음처럼 확장한다.

```text
사건 개요
음성/영상 자료
카톡/문자 캡처
문서 자료
녹취록 의뢰
납품 문서
관리자 메모
진행 상태
```

이 방향으로 잡으면 녹취록 플랫폼에서 출발해 `사건 증거자료 정리 플랫폼`으로 확장할 수 있다.
